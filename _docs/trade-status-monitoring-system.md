# Система мониторинга статусов трейдов

## ЛОГИКА СИСТЕМЫ ПРОВЕРКИ СТАТУСОВ ТРЕЙДОВ

### 1. ЭТАПЫ ЖИЗНЕННОГО ЦИКЛА ТРЕЙДА

**Этап 1: Создание трейда**
- Заказ переходит из статуса `paid` в `processing`
- Создается TradeOffer со статусом `Pending`
- Отправляется запрос в Steam API для создания трейда
- Steam возвращает статус `Active` (готов к принятию) или `CreatedNeedsConfirmation` (нужно подтверждение продавца)

**Этап 2: Ожидание подтверждения продавца (если требуется)**
- TradeOffer находится в статусе `CreatedNeedsConfirmation`
- Продавец должен подтвердить трейд в Steam клиенте или мобильном приложении
- После подтверждения статус меняется на `Active`
- Заказ остается в статусе `processing`

**Этап 3: Ожидание действий покупателя**
- TradeOffer находится в статусе `Active`
- Покупатель видит трейд и может его принять или отклонить
- Заказ остается в статусе `processing`

**Этап 4: Завершение трейда**
- Покупатель принимает трейд → статус `Accepted`
- Покупатель отклоняет трейд → статус `Declined`
- Трейд истекает по времени → статус `Expired`
- Продавец отменяет трейд → статус `Canceled`

**Этап 5: Финализация заказа**
- При `Accepted`: проверяем фактическую передачу предметов и завершаем заказ
- При других статусах: отменяем заказ и возвращаем средства

### 2. СИСТЕМА ПЕРИОДИЧЕСКОЙ ПРОВЕРКИ

**Частота проверок:**
- Активные трейды (`Active`, `CreatedNeedsConfirmation`) - каждые 5 минут
- Принятые трейды (`Accepted`) - каждый час до подтверждения передачи
- Все остальные статусы обрабатываются немедленно

**Приоритизация:**
- Сначала проверяются трейды с истекающим сроком резерва заказа
- Затем недавно созданные трейды
- Потом все остальные по дате создания

**Ограничения нагрузки:**
- Максимум 10 одновременных запросов к Steam API
- Пауза между запросами для одного пользователя
- Retry механизм при ошибках Steam API

### 3. ЛОГИКА ПРОВЕРКИ ЗАВЕРШЕНИЯ ТРЕЙДА

**Проблема холда предметов:**
- Принятые предметы попадают в холд на срок до 7 дней
- Steam API не показывает предметы в холде как торгуемые
- Стандартный inventory API не отображает предметы в холде

**Стратегия проверки передачи предметов:**

**Шаг 1: Базовая проверка**
- Получаем полный инвентарь покупателя через Steam API
- Ищем asset_id предметов из трейда в инвентаре покупателя
- Если все найдены - трейд успешно завершен

**Шаг 2: Проверка через историю трейдов**
- Если предметы не найдены в инвентаре, запрашиваем историю трейдов покупателя
- Проверяем, есть ли в истории наш trade_offer_id с успешным статусом
- Проверяем совпадение asset_id в истории трейдов

**Шаг 3: Временная логика**
- Если трейд был принят менее 24 часов назад - откладываем финальную проверку
- Ждем до 7 дней для выхода предметов из холда
- После 7 дней делаем окончательную проверку

**Шаг 4: Проверка через продавца**
- Проверяем, что предметы исчезли из инвентаря продавца
- Сверяем asset_id - они должны отсутствовать у продавца

### 4. ОБНОВЛЕНИЕ СТАТУСОВ ЗАКАЗОВ

**При статусе `Active` или `CreatedNeedsConfirmation`:**
- Заказ остается в `processing`
- Продолжаем мониторинг
- Проверяем не истек ли срок резерва заказа

**При статусе `Accepted`:**
- Запускаем процедуру проверки передачи предметов
- Если передача подтверждена - меняем заказ на `completed`
- Если не подтверждена - ждем и перепроверяем

**При статусе `Declined`, `Canceled`, `Expired`:**
- Немедленно меняем заказ на `cancelled`
- Возвращаем средства покупателю
- Освобождаем предметы продавца

### 5. ОБРАБОТКА ОСОБЫХ СЛУЧАЕВ

**Ошибки Steam API:**
- При временных ошибках (500, 503) - повторяем через возрастающие интервалы
- При ошибках авторизации - помечаем продавца как требующего повторной авторизации
- При неизвестных ошибках - логируем и уведомляем администраторов

**Несоответствие данных:**
- Если Steam API показывает несуществующий статус - логируем и используем fallback
- Если asset_id изменились - проверяем по classid/instanceid
- Если данные противоречивы - приоритет у более свежих данных

**Таймауты и лимиты:**
- Каждый запрос к Steam API имеет таймаут 15 секунд
- При превышении rate limit - откладываем проверки на период охлаждения
- Максимальное время ожидания завершения трейда - 7 дней

**Мониторинг и алерты:**
- Уведомления при критических ошибках Steam API
- Статистика успешности проверок
- Алерты при аномально долгих трейдах
- Мониторинг времени отклика Steam API

### 6. ИНТЕГРАЦИЯ С СУЩЕСТВУЮЩЕЙ СИСТЕМОЙ

**Связь с отменой заказов:**
- При ручной отмене заказа также отменяем соответствующий трейд
- При автоматической отмене по истечению резерва также отменяем трейд
- Синхронизируем статусы между Order и TradeOffer

**Обновление UI в реальном времени:**
- WebSocket уведомления об изменении статусов
- Автообновление страниц заказов при изменении статуса
- Push-уведомления мобильному приложению (если есть)

**Логирование и аудит:**
- Все изменения статусов логируются с временными метками
- Сохраняем snapshot данных Steam API для расследований
- Ведем метрики по времени завершения трейдов

## АРХИТЕКТУРА СИСТЕМЫ

### Основные компоненты:

1. **TradeStatusChecker** - основной сервис проверки статусов
2. **Enhanced InventoryService** - расширенный сервис инвентаря с поддержкой холда
3. **CheckTradeStatusesJob** - джоб для периодических проверок
4. **TradeCompletionVerifier** - верификатор завершения трейдов

### Схема работы:

```
Scheduler → CheckTradeStatusesJob → TradeStatusChecker → Steam API
                                                      ↓
Enhanced InventoryService ← TradeCompletionVerifier ← Trade Status Response
                         ↓
                    Order Status Update → WebSocket → Frontend
```

### Расписание:

- **CheckTradeStatusesJob**: каждые 5 минут для активных трейдов
- **CheckAcceptedTradesJob**: каждый час для принятых трейдов
- **TradeTimeoutJob**: ежедневно для очистки просроченных трейдов
